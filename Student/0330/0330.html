⭐ 1. 高频 Follow-up Questions（必问）

❓ Q1: Why not trigger Lambda directly from SNS?

✅ Answer

SNS can trigger Lambda directly, but using SQS in between provides buffering, retry capability, and decoupling. This prevents Lambda from being overwhelmed during traffic spikes.

👉 关键词：

  • buffering
  • decoupling
  • reliability

❓ Q2: What happens if Lambda fails?

✅ Answer

SQS retains the message and retries processing. We can also configure a Dead Letter Queue (DLQ) to capture failed messages after multiple retries.

👉 关键词：

  • retry
  • DLQ
  • fault tolerance

❓ Q3: Why use multiple SQS queues instead of one?

✅ Answer

Multiple queues allow independent processing, better isolation, and different scaling strategies for each workload.

👉 关键词：

  • isolation
  • scalability
  • separation of concerns

❓ Q4: How does filtering work in SNS?

✅ Answer

SNS filtering allows messages to be routed to specific subscribers based on message attributes or message body, ensuring that each consumer only receives relevant messages.

👉 关键词：

  • selective delivery
  • efficiency

❓ Q5: How would you debug if no Lambda logs appear?

✅ Answer

If no logs appear in CloudWatch, it means Lambda was not invoked. I would trace upstream: check SQS triggers, SNS subscriptions, and message filtering.

👉 关键词：

  • trace upstream
  • CloudWatch

❓ Q6: How do you handle high traffic?

✅ Answer

SQS automatically scales to handle large volumes, and Lambda can scale horizontally. We can also control concurrency and batch size.

👉 关键词：

  • horizontal scaling
  • concurrency
  • batching

❓ Q7: What is the benefit of event-driven architecture?

✅ Answer

It improves decoupling, scalability, and allows asynchronous processing, making the system more flexible and resilient.

👉 关键词：

  • async
  • decoupling
  • scalability

❓ Q8: What if message order matters?

✅ Answer

We would use FIFO SQS queues to guarantee ordering and exactly-once processing.

👉 关键词：

  • FIFO
  • ordering

🎯 3. 面试核心对比（你要会说）

🔹 SNS vs SQS

SNS is used for message distribution (fan-out), while SQS is used for buffering and decoupling.

🔹 SNS + SQS together

SNS distributes messages, and SQS ensures reliable delivery and buffering before processing.

🔹 Kafka vs SNS/SQS

Kafka is designed for high-throughput streaming and message replay, while SNS/SQS are simpler managed services for event-driven workflows.

🔹 EventBridge vs SNS

EventBridge provides more advanced event routing with rules and integrates deeply with AWS services.

SNS handles message fan-out, SQS provides buffering and reliability, and Lambda processes messages asynchronously. Compared to Kafka, this solution is simpler and fully managed, while Kafka is better suited for high-throughput streaming and replay scenarios.

🧠 Dive Deep – Complex Issue STAR Story (Interview Ready 🔥)

⭐ Situation

During an internship, I worked on an AWS-based event-driven pipeline using SNS, SQS, and Lambda.
Under production-like load, we noticed intermittent message loss—some messages were successfully published but never processed by downstream Lambda functions.

⭐ Task

I was responsible for identifying the root cause of the message loss and ensuring reliable, end-to-end delivery in the pipeline.

⭐ Action

I approached the issue systematically and dove deep into the system:

  • Step 1: Validate symptom

    • Compared SNS publish count vs Lambda invocation count
    • Confirmed mismatch → real message loss

  • Step 2: Break down pipeline

    • SNS → SQS → Lambda
    • Checked each stage independently

  • Step 3: Investigate SQS

    • Monitored queue metrics (ApproximateNumberOfMessages)
    • Found messages were entering SQS but not always consumed

  • Step 4: Analyze Lambda triggers

    • Checked batch size, visibility timeout, and concurrency
    • Discovered visibility timeout < processing time → messages reappeared

  • Step 5: Inspect error handling

    • Found some messages failed silently without retry tracking
    • No DLQ (Dead Letter Queue) configured

  • Step 6: Fix design

    • Increased visibility timeout
    • Added DLQ for failed messages
    • Enabled CloudWatch alarms for failure metrics
    • Improved logging for each processing step

⭐ Result

  • Eliminated message loss (100% delivery observed in testing)
  • Improved system reliability and observability
  • Reduced debugging time for future issues by adding DLQ and metrics

🎯 One-Line Summary

I identified message loss by comparing system metrics, traced the pipeline stage by stage, and fixed visibility timeout and error handling issues to ensure reliable message processing.

🧠 10 Follow-up Questions + Reference Answers (Amazon Style 🔥)

1. How did you confirm the issue was real?

Answer:
I compared SNS publish metrics with Lambda invocation metrics in CloudWatch and found a consistent mismatch, confirming message loss.

2. Why did you check metrics first?

Answer:
Metrics give objective evidence and help validate whether the issue is real before diving into assumptions.

3. How did you isolate the problem?

Answer:
I broke the pipeline into stages (SNS → SQS → Lambda) and verified each stage independently to identify where messages were lost.

4. What was the root cause?

Answer:
The visibility timeout was shorter than the Lambda processing time, causing messages to reappear and sometimes be dropped or duplicated.

5. Why didn’t you catch this earlier?

Answer:
Initial testing used low load and short processing times, which didn’t expose the timing issue under real conditions.

6. How did you fix the issue?

Answer:
I increased the visibility timeout, added a Dead Letter Queue, and improved logging and monitoring.

7. What trade-offs did you consider?

Answer:
A longer visibility timeout reduces duplicate processing but increases retry delay, so I balanced it based on average processing time.

8. How did you validate your fix?

Answer:
I re-ran load tests and verified that SNS publish count matched Lambda invocation count, with no message loss.

9. What did you learn from this experience?

Answer:
I learned the importance of understanding system-level timing and using metrics to guide debugging instead of guessing.

10. How would you prevent this in the future?

Answer:
I would add monitoring dashboards, alarms, DLQs by default, and test under production-like load earlier in development.
