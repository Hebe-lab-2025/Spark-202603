这张图在讲 AWS CDK 和 AWS CloudFormation 的关系，可以这样理解：


## 一、共同点

* 两者**都是用来做基础设施即代码（Infrastructure as Code, IaC）**的
  👉 也就是用代码来创建云资源（如服务器、数据库、网络等）

## 二、核心区别

### 1️⃣ 编写方式不同

* **CDK**

  * 用编程语言写（更“开发者友好”）
  * 支持：TypeScript、JavaScript、Python、Java、.NET 等
  * 可以用变量、循环、函数等编程能力

* **CloudFormation**

  * 用**YAML / JSON 模板**写
  * 更偏“配置文件”，不是真正编程语言
  * 复杂逻辑写起来比较繁琐


### 2️⃣ 工作原理（关键点 ⭐）

* CDK **不是直接部署资源**
* 它会先执行一个步骤：`cdk synth`

👉 这个过程会把 CDK 代码：
➡️ **转换（编译）成 CloudFormation 模板（YAML/JSON）**

然后：
➡️ 再由 CloudFormation 去真正创建资源

📌 一句话总结：

> **CDK 是 CloudFormation 的“上层封装”**


## 三、关系总结（最重要）

可以这样理解它们的关系：

```
你写 CDK 代码
     ↓
cdk synth（生成模板）
     ↓
CloudFormation 模板
     ↓
CloudFormation 部署资源
```

## 四、为什么用 CDK？

因为它继承了 CloudFormation 的所有能力：

* ✅ 安全部署（不会乱改资源）
* ✅ 自动回滚（失败会恢复）
* ✅ 漂移检测（drift detection）
* ✅ 资源依赖管理

同时又更好用：

* 👍 更少重复代码
* 👍 更强表达能力
* 👍 更适合工程化开发

## 五、简单类比帮助理解

* CloudFormation 👉 **“施工蓝图（YAML/JSON）”**
* CDK 👉 **“用代码生成蓝图的工具”**

好，我们用一个**最简单但典型的例子：创建一个 S3 存储桶**来对比 AWS CDK 和 AWS CloudFormation。


# 🧪 示例目标

创建一个：

* S3 Bucket
* 开启版本控制（versioning）

# 一、CloudFormation 写法（YAML）

```yaml
Resources:
  MyBucket:
    Type: AWS::S3::Bucket
    Properties:
      VersioningConfiguration:
        Status: Enabled
```

👉 特点：

* 纯配置
* 结构固定
* 没有变量、循环、逻辑能力（很弱）


# 二、CDK 写法（TypeScript）

```typescript
import * as cdk from 'aws-cdk-lib';
import * as s3 from 'aws-cdk-lib/aws-s3';

export class MyStack extends cdk.Stack {
  constructor(scope: cdk.App, id: string) {
    super(scope, id);

    new s3.Bucket(this, 'MyBucket', {
      versioned: true
    });
  }
}
```

👉 特点：

* 用真实编程语言写
* 可以用变量、函数、循环
* 可读性更强


# 三、CDK 背后发生了什么（关键 ⭐）

当你运行：

```bash
cdk synth
```

CDK 会生成类似这样的 CloudFormation：

```yaml
Resources:
  MyBucketF68F3FF0:
    Type: AWS::S3::Bucket
    Properties:
      VersioningConfiguration:
        Status: Enabled
```

> **CDK = 帮你自动写 CloudFormation**


# 四、再看一个更真实的优势（对比差距）
## 🎯 场景：创建多个 Bucket：
### ❌ CloudFormation（很笨）

```yaml
Resources:
  Bucket1:
    Type: AWS::S3::Bucket

  Bucket2:
    Type: AWS::S3::Bucket

  Bucket3:
    Type: AWS::S3::Bucket
```

### ✅ CDK（优雅很多）

```typescript
const bucketNames = ['a', 'b', 'c'];

for (const name of bucketNames) {
  new s3.Bucket(this, `Bucket-${name}`);
}
```

👉 编程能力直接碾压

# 五、总结一句话

👉 AWS CloudFormation= **底层执行引擎（真正创建资源）**

👉 AWS CDK = **更高级、更好用的“写模板工具”**


# 六、什么时候选哪个？

* 想**简单、直接、标准化** 👉 用 CloudFormation
* 想**开发效率高、逻辑复杂、工程化** 👉 用 CDK（主流推荐）

## 一、整体结构（从大到小）
可以把 AWS CDK 理解成一个“搭积木”的体系：

### 1️⃣ App（应用）
* 最外层容器，相当于整个项目入口
* 一个 App 可以包含多个 Stack
* 就像一个程序的 main 函数

👉 支持语言：
* TypeScript、JavaScript、Python、Java、C# 等

---

### 2️⃣ Stack（栈）
* App 里面的“部署单元”
* 一个 Stack 最终会变成一个 AWS CloudFormation 模板

👉 可以理解为：
* 一组相关资源的集合（比如一个后端服务）

### 3️⃣ Construct（构造）
* Stack 里的“组件”
* 用来封装具体功能或资源组合

👉 例如图中：
* 一个 Construct 里包含：
  * Amazon SQS 队列
  * AWS Lambda 函数

* 另一个 Construct 里包含：
  * Amazon S3 存储桶
  * Amazon DynamoDB 数据表

👉 核心思想：
> 把复杂资源打包成可复用模块

### 4️⃣ Resources（资源）
* 最底层，真正的 AWS 资源
* 例如：
  * S3 Bucket
  * Lambda Function
  * DynamoDB Table
👉 这些才是最终在 AWS 上运行的东西

## 二、执行流程（图中箭头）
流程是这样的👇
1. 你用 CDK 写代码（App / Stack / Construct）
2. CDK 自动生成 CloudFormation 模板
3. CloudFormation 帮你创建资源
4. 最终资源部署到 AWS 上

👉 简单一句话：
> CDK代码 → CloudFormation → AWS资源

## 三、核心理解（考试/面试重点）
你可以记住这个层级关系：

```
App
 └── Stack
      └── Construct
           └── Resource
```

## 四、通俗类比
用“搭乐高”来理解：
* App = 整个乐高项目
* Stack = 一个场景（比如房子）
* Construct = 房子的模块（门、窗、厨房）
* Resource = 具体积木块

## 五、为什么用 CDK？
相比直接写 CloudFormation：
✅ 用编程语言（更灵活）
✅ 可以复用代码（Construct）
✅ 更容易维护大型项目
