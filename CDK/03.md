## 🧱 整体理解

一个 AWS CDK 项目，本质就是：
👉 用代码（TypeScript / Python 等）来定义云资源
👉 最终转换成 AWS CloudFormation 模板去部署


## 📁 关键文件讲解（图中编号）

### **1️⃣ `cdk.json`**

👉 作用：告诉 CDK CLI “怎么运行你的程序”

* 类似启动配置文件
* 指定入口命令，比如：

  ```bash
  npx ts-node bin/xxx.ts
  ```

📌 可以理解为：
➡️ “项目启动说明书”

### **2️⃣ `bin/cdk-s3.ts`（入口文件）**
👉 CDK 应用的入口（main函数）

它做的事情：
* 创建 App
* 加载 Stack

示例逻辑：
```ts
const app = new cdk.App();
new MyStack(app, 'MyStack');
```

📌 类比：
➡️ 程序的 main()

### **3️⃣ `lib/cdk-s3-stack.ts` ⭐（最重要）**
👉 核心文件！！！

这里定义：
* AWS 资源（S3、Lambda、DynamoDB等）
* 架构逻辑

示例：
```ts
new s3.Bucket(this, 'MyBucket');
```

📌 类比：
➡️ “真正干活的地方”
➡️ 你写的所有云资源都在这里

### **4️⃣ `node_modules/`**

👉 项目依赖（自动生成）
📌 不用管：
* npm install 自动生成
* 存放 CDK 库

### **5️⃣ `test/`**

👉 单元测试代码

* 用来测试你的 stack 是否正确
* 可选，但企业项目常用

### **6️⃣ `.gitignore`**

👉 Git 忽略文件列表

📌 防止上传：

* node_modules
* 临时文件

### **7️⃣ `package.json`**
👉 Node.js 项目配置

包含：
* 依赖（dependencies）
* 脚本（scripts）

📌 类比：
➡️ 项目的“说明书 + 依赖清单”

### **8️⃣ `tsconfig.json`**
👉 TypeScript 编译配置

控制：
* 编译规则
* 输出格式

## 🧠 一句话总结整个结构

👉 **bin → 启动入口**
👉 **lib → 定义资源（核心）**
👉 **cdk.json → 告诉 CDK 怎么运行**
👉 **其他 → 辅助工具文件**


## 🚀 小总结（考试/面试版）

* CDK 项目由 **App → Stack → Construct** 组成
* `bin/` 是入口
* `lib/` 是核心业务逻辑
* `cdk.json` 控制运行方式
* 最终生成 CloudFormation 模板并部署

## 📁 一、整体结构（AWS CDK 项目）

一个典型的 AWS CDK 项目一般分成两部分：

* **`bin/`** 👉 程序入口（启动整个 CDK 应用）
* **`lib/`** 👉 定义具体的 AWS 资源（Stack）

---

## 🔢 二、代码中每个编号的含义

### **1️⃣ `import 'source-map-support/register'`**

* 用来提升**报错信息的可读性**
* 可以把 JS 错误映射回 TypeScript 源码
* 👉 方便调试

---

### **2️⃣ `import * as cdk from 'aws-cdk-lib'`**

* 引入 CDK 的核心库
* 里面包含：

  * `App`（应用）
  * `Stack`（栈）
  * 各种 AWS 资源构造类

👉 相当于“基础工具箱”

---

### **3️⃣ `import { CdkS3Stack } from '../lib/cdk-s3-stack'`**

* 从 `lib/` 文件夹导入你自己写的 Stack
* 这个 Stack 里定义了具体资源（比如 S3）

👉 相当于“加载你写的基础设施代码”

---

### **4️⃣ `const app = new cdk.App();`**

* 创建一个 **CDK 应用实例**
* 👉 整个程序的入口

可以理解为：

> “我要开始一个 CDK 项目了”

---

### **5️⃣ `new CdkS3Stack(app, 'CdkS3Stack', {...})`**

* 创建（实例化）一个 Stack

参数说明：

* `app` 👉 这个 stack 属于哪个应用

# 一、S3 Bucket 的 Props（属性）

在 CDK 里创建资源时，会传入一个“配置对象”（props），这里是 S3 Bucket 的常见参数：

### 1. `bucketName?`

* 👉 桶的名字（可选）
* 示例：

  ```ts
  bucketName: 'my-bucket-123'
  ```
* ⚠️ 注意：

  * 全局唯一（全世界不能重复）
  * 如果不写，AWS 会自动生成

---

### 2. `versioned?`

* 👉 是否开启版本控制（true / false）
* 示例：

  ```ts
  versioned: true
  ```
* 📌 作用：

  * 文件删除/覆盖后还能恢复旧版本
* ✅ 常用于：

  * 数据备份
  * 防误删

---

### 3. `publicReadAccess?`

* 👉 是否允许公开读取
* 示例：

  ```ts
  publicReadAccess: true
  ```
* ⚠️ 注意：

  * 一般**生产环境不建议开启**
  * 可能导致数据泄露

---

# 二、Logical ID（逻辑 ID）

* 👉 CDK/CloudFormation 内部使用的名字

* 示例：

  ```ts
  new s3.Bucket(this, 'MyBucket', { ... })
  ```

  👉 `'MyBucket'` 就是 Logical ID

* 📌 特点：

  * 在代码里用
  * 用于 CloudFormation 识别资源
  * 不等于真实资源名

---

# 三、Physical ID（物理 ID）

* 👉 AWS 实际创建的资源名字（真实存在的）

比如：

* S3 bucket 名
* Lambda 名
* DynamoDB 表名

📌 关系：

| 类型          | 含义         |
| ----------- | ---------- |
| Logical ID  | CDK/模板里的名字 |
| Physical ID | AWS 实际资源名字 |

---

# 四、总结一句话理解

👉 CDK 创建资源 =

```
new 资源(this, LogicalID, Props)
```

* `this` → 作用域（Scope）
* `LogicalID` → 模板里的名字
* `Props` → 配置参数（bucketName / versioned 等）

---

# 五、完整例子（帮助你理解）

```ts
const bucket = new s3.Bucket(this, 'MyBucket', {
  bucketName: 'my-demo-bucket-123',
  versioned: true,
  publicReadAccess: false
});
```

👉 含义：

* 创建一个 S3 Bucket
* 名字是 `my-demo-bucket-123`
* 开启版本控制
* 不公开访问

# 一、AWS DynamoDB Props（属性）

创建 DynamoDB 表时，一般会写：

```ts
new dynamodb.Table(this, 'TableId', {
  // props 在这里
})
```

---

## 1. `readCapacity?`（读容量）

* 👉 表示每秒可以执行多少次“读操作”
* 单位：RCU（Read Capacity Units）

### 示例：

```ts
readCapacity: 5
```

### 📌 含义：

* 每秒支持约 5 次强一致读（或更多弱一致读）

### ⚠️ 注意：

* 只在 **Provisioned 模式（预置容量）** 下使用
* On-Demand 模式不需要设置

---

## 2. `writeCapacity?`（写容量）

* 👉 每秒可以执行多少次“写操作”
* 单位：WCU（Write Capacity Units）

### 示例：

```ts
writeCapacity: 5
```

### 📌 含义：

* 每秒支持 5 次写操作

---

## 3. `partitionKey`（分区键）⭐最重要

* 👉 DynamoDB 表的主键（必须有）

### 示例：

```ts
partitionKey: {
  name: 'id',
  type: dynamodb.AttributeType.STRING
}
```

### 📌 作用：

* 用来唯一标识数据
* 决定数据如何分布（分区）

### 常见类型：

* STRING
* NUMBER
* BINARY

---

## 4. `tableName?`

* 👉 表的名字（可选）

### 示例：

```ts
tableName: 'my-table'
```

### ⚠️ 注意：

* 不写的话 AWS 自动生成
* 写了必须全局唯一（同账号同区域）

---

# 二、补充：实际开发更常用配置

👉 实际项目中通常还会加👇

### 1. 计费模式（推荐）

```ts
billingMode: dynamodb.BillingMode.PAY_PER_REQUEST
```

👉 好处：

* 不用设置 read/write capacity
* 按请求付费（更简单）

---

### 2. 排序键（可选）

```ts
sortKey: {
  name: 'createdAt',
  type: dynamodb.AttributeType.STRING
}
```

👉 用于：

* 排序
* 组合主键（partition + sort）

---

# 三、完整示例（推荐写法）

```ts
const table = new dynamodb.Table(this, 'MyTable', {
  tableName: 'demo-table',

  partitionKey: {
    name: 'id',
    type: dynamodb.AttributeType.STRING
  },

  billingMode: dynamodb.BillingMode.PAY_PER_REQUEST
});
```

---

# 四、一句话总结

👉 DynamoDB 必须记住三件事：

* **partitionKey**（必须有）
* **capacity 或 billingMode**
* **tableName（可选）**

# 一、什么是 IAM Role（给 Lambda 用）

Lambda 想访问：

* Amazon S3
* Amazon DynamoDB
* Amazon CloudWatch

👉 都必须通过 Role 授权

# 二、IAM Role Props 详解

创建方式：

```ts
new iam.Role(this, 'RoleId', {
  // props
})
```

---

## 1. `roleName?`

* 👉 角色名称（可选）

```ts
roleName: 'my-lambda-role'
```

📌 特点：

* 不写 → AWS 自动生成
* 写了 → 必须唯一

---

## 2. `description?`

* 👉 描述这个角色是干嘛的

```ts
description: 'Role for Lambda to access S3 and DynamoDB'
```

📌 作用：

* 方便团队理解
* 不影响功能（只是说明）

---

## 3. `assumedBy` ⭐最重要

* 👉 谁可以“使用这个角色”

```ts
assumedBy: new iam.ServicePrincipal('lambda.amazonaws.com')
```

📌 含义：

* 允许 Lambda 使用这个 Role

👉 常见值：

| 服务     | 写法                        |
| ------ | ------------------------- |
| Lambda | `lambda.amazonaws.com`    |
| EC2    | `ec2.amazonaws.com`       |
| ECS    | `ecs-tasks.amazonaws.com` |

---

## 4. `managedPolicies?`

* 👉 AWS 已经写好的权限策略（推荐用）

```ts
managedPolicies: [
  iam.ManagedPolicy.fromAwsManagedPolicyName('service-role/AWSLambdaBasicExecutionRole')
]
```

📌 常见策略：

### Lambda 基础日志权限

* `AWSLambdaBasicExecutionRole`
  👉 允许写日志到 CloudWatch

---

# 三、完整示例（标准写法）

```ts
const role = new iam.Role(this, 'MyLambdaRole', {
  roleName: 'demo-lambda-role',

  assumedBy: new iam.ServicePrincipal('lambda.amazonaws.com'),

  managedPolicies: [
    iam.ManagedPolicy.fromAwsManagedPolicyName(
      'service-role/AWSLambdaBasicExecutionRole'
    )
  ]
});
```

* **managedPolicies** → 给权限
* **roleName** → 可选名字

# 📊 什么是 CloudWatch Alarm？

Amazon CloudWatch 是 AWS 的监控服务。

👉 **Alarm（告警）**的作用：
当某个指标异常时，自动提醒你（比如报错太多）

# ⚡ 场景（重点理解）

比如你有一个
👉 AWS Lambda 函数

你希望：

> ❗ 如果错误次数太多 → 立刻报警

# 🧩 关键属性（你图里的）

## 1️⃣ evaluationPeriods

👉 连续多少个周期满足条件才触发报警

例：

* `evaluationPeriods: 2`
  👉 连续 2 次都超标才报警（避免误报）


## 2️⃣ threshold

👉 阈值（触发条件）

例：

* `threshold: 5`
  👉 错误次数 > 5 就报警


## 3️⃣ alarmName

👉 报警名字（可选）

例：

```ts
alarmName: 'LambdaErrorAlarm'
```

---

## 4️⃣ metric（核心🔥）

```ts
metric: lambdaFunction.metricErrors()
```

👉 含义：
监控 Lambda 的 **错误次数**

# 🧪 完整 CDK 示例（非常重要🔥）

```ts
import * as cloudwatch from 'aws-cdk-lib/aws-cloudwatch';
import * as lambda from 'aws-cdk-lib/aws-lambda';

// 假设你已经有一个 Lambda
const fn = new lambda.Function(this, 'MyFunction', {
  runtime: lambda.Runtime.NODEJS_18_X,
  handler: 'index.handler',
  code: lambda.Code.fromInline('exports.handler = async () => {};')
});

// 创建 Alarm
new cloudwatch.Alarm(this, 'LambdaErrorAlarm', {
  metric: fn.metricErrors(),
  threshold: 5,
  evaluationPeriods: 1,
  alarmName: 'LambdaErrorAlarm'
});
```

# 🚨 报警触发逻辑（考试重点）

👉 条件：

```
错误次数 > threshold
并且连续 evaluationPeriods 次满足
```

才会报警
