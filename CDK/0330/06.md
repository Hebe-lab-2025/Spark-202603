# STAR Story 1 — Build cloud infrastructure with CDK

## Situation

Our team wanted to build a small AWS-based event-driven application, but manually creating resources in the AWS Console would be slow, error-prone, and hard to reproduce.

## Task

I needed to define the infrastructure in a scalable and repeatable way, and also speed up development so we could focus more on logic instead of boilerplate setup.

## Action

I used AWS CDK to define resources such as:

* API Gateway
* Lambda functions
* SQS queue
* S3 bucket

To improve development speed, I also used CodeWhisperer in VS Code to generate AWS CDK code suggestions and Lambda boilerplate.

For example, it helped generate:

* resource definitions
* IAM permission setup
* event handling templates

I reviewed the generated code carefully, adjusted naming and configuration, and deployed the full stack using `cdk deploy`.

## Result

We were able to provision the full environment quickly and consistently.
The infrastructure became version-controlled, easier to update, and much more repeatable than manual console setup.
Using CodeWhisperer also reduced boilerplate coding time and helped us move faster.

---

# STAR Story 2 — Use CodeWhisperer to improve productivity

## Situation

When building AWS projects, a lot of time can be spent writing repetitive infrastructure code and Lambda templates.

I wanted to reduce manual coding effort while still keeping code quality and correctness.

I used CodeWhisperer to generate starter code for:

* CDK resource definitions

Instead of accepting everything directly, I treated it as a productivity assistant.
I verified the generated code, checked resource configuration, and tested the deployment flow end to end.

## Q4. What are the risks of using AI-generated code?

**Answer:**

The biggest risk is accepting code without review.
AI-generated code may be syntactically correct but still have wrong assumptions, unnecessary permissions, or inefficient logic, so manual validation is still necessary.

## Q7. How do you deploy a CDK project?

# 💻 5️⃣ CDK + Lambda + SQS 示例代码（Java版🔥）

👉 面试官最爱这个

## 📦 CDK Stack

```java
public class MyStack extends Stack {

    public MyStack(final Construct scope, final String id) {
        super(scope, id);

        // 1️⃣ Create SQS Queue
        Queue queue = Queue.Builder.create(this, "MyQueue")
                .build();

        // 2️⃣ Create Lambda Function
        Function function = Function.Builder.create(this, "MyLambda")
                .runtime(Runtime.JAVA_11)
                .handler("com.example.Handler::handleRequest")
                .code(Code.fromAsset("lambda"))
                .build();

        // 3️⃣ Grant Lambda permission to read SQS
        queue.grantConsumeMessages(function);

        // 4️⃣ Trigger Lambda from SQS
        function.addEventSource(new SqsEventSource(queue));
    }
}
```

---

## ⚙️ Lambda Handler

```java
public class Handler implements RequestHandler<String, String> {

    @Override
    public String handleRequest(String input, Context context) {
        System.out.println("Received: " + input);
        return "Processed: " + input;
    }
}
```

# 4. Failure Handling 版本（面试必杀）

## 4.1 如果 Lambda 处理失败怎么办

标准回答：

* use retries
* configure DLQ
* log failed payloads
* alert through CloudWatch alarms

你可以这样说：

> If downstream processing failed, I would rely on SQS retry behavior and configure a dead-letter queue to isolate poison messages. That prevents a single bad message from blocking the whole pipeline.

---

## 4.2 什么是 DLQ

**DLQ = Dead Letter Queue**

作用：

* 存放多次处理失败的消息
* 避免坏消息一直重试
* 方便后续排查和补偿

面试一句话：

> DLQ helps isolate messages that repeatedly fail processing, so the main queue can continue moving normally.

---

## 4.3 Retry 怎么讲

> I would use controlled retries for transient failures, such as temporary downstream service errors, but avoid infinite retries for permanent data issues.

这个回答很好，因为你区分了：

* transient failure：临时故障，可重试
* permanent failure：数据坏了，不该无限重试

---

## 4.4 Idempotency 怎么讲

这是高频必问。

因为消息系统里，**同一条消息可能会被重复处理**。

你可以这样答：

> Since SQS can deliver messages more than once, I would make the consumer idempotent. For example, I could attach a unique request ID and store processed IDs in DynamoDB, so duplicate messages would not create duplicate side effects.

中文理解：

* 给每条请求一个唯一 ID
* 处理前先检查是否处理过
* 避免重复写库、重复扣费、重复发通知

---

## 4.5 Partial Failure 怎么讲

如果系统有多个步骤，一个步骤成功，一个失败，怎么办？

你可以这样说：

> I would separate the workflow into clear stages and persist status updates, so failures are visible and recoverable. That makes the system easier to debug and resume safely.

这个回答说明你考虑了：

* 状态管理
* 可恢复性
* 可观测性

# 6. Amazon 风格的高频 follow-up + 参考答案

## Q1. Why did you introduce SQS instead of calling Lambda synchronously?

> I wanted to decouple request intake from downstream processing. That improved reliability, absorbed traffic spikes, and reduced the chance that a slow downstream component would impact the user-facing API.

## Q2. How would you scale the system?

> Lambda scales automatically, and SQS helps buffer bursts. I would monitor queue depth, Lambda concurrency, and processing latency to make sure the consumer keeps up with demand.

## Q3. What happens if the same message is processed twice?

> I would design the consumer to be idempotent, usually by using a unique request ID and checking whether the operation was already completed before applying side effects.

## Q4. How do you handle poison messages?

> I would use a dead-letter queue after a defined retry threshold, so invalid or repeatedly failing messages do not block the rest of the pipeline.

## Q5. How do you know there is a bottleneck?

> I would look at CloudWatch metrics such as API latency, Lambda duration, error count, queue depth, and age of oldest message. Those metrics make bottlenecks visible quickly.

## Q6. What are the trade-offs of serverless?

> The main benefits are low operational overhead and easy scaling. The trade-offs include cold starts, concurrency limits, and less control compared with long-running services.

## Q7. Why use CDK?

> CDK improves repeatability, maintainability, and collaboration because infrastructure is defined in code, version-controlled, and deployable across environments.

## Q8. What did CodeWhisperer help with the most?

> It was most useful for generating CDK resource patterns, Lambda handler templates, and AWS SDK boilerplate, which let me focus more on design and correctness.

## Q9. How would you deploy safely to production?

> I would use multiple environments such as dev, staging, and prod, validate infrastructure changes through CI/CD, and monitor deployment health using logs, alarms, and rollback strategies.

## Q10. What would you improve next?

> I would add stronger observability, better retry policies, idempotency enforcement, and more structured status tracking in DynamoDB for easier debugging and recovery.

# AWS CDK and CodeWhisperer 中文讲解

## 1. 这页到底在讲什么？

这页的核心意思是：

> 我们可以用 **AWS CDK** 来“用代码创建云资源”，再配合 **CodeWhisperer** 这种 AI 编码助手，更快地把 AWS 架构搭出来。

这张图里搭出来的资源主要有两个：

* **Amazon RDS**
* **Amazon S3**

其中：

* **RDS** 放在 **VPC** 里面
* **S3** 放在 AWS Cloud 里，但不属于 VPC 内部资源

---

# 2. 图里的每个东西分别是什么？

## AWS Cloud

就是整个 AWS 云环境。

你可以理解成：

* 这是一个大的云平台
* 你的所有资源都部署在这里
* 比如数据库、存储、计算服务都在 AWS Cloud 里面

---

## VPC = Virtual Private Cloud

中文一般叫：

> **虚拟私有云**

你可以把它理解成：

* AWS 里给你划出来的一块“私有网络”
* 你可以自己控制 IP、子网、安全组、路由
* 适合放敏感资源，比如数据库

### 为什么 RDS 常放在 VPC 里？

因为数据库通常不希望直接暴露给公网。

所以放进 VPC 的好处是：

* 更安全
* 可以限制谁能访问
* 更方便和 EC2 / Lambda / ECS 等内部服务通信

### 面试英文说法

> A VPC provides network isolation and allows us to securely place resources such as databases in private subnets.

---

## Amazon RDS

RDS = **Relational Database Service**

中文可以说：

> AWS 托管的关系型数据库服务

它支持：

* MySQL
* PostgreSQL
* MariaDB
* Oracle
* SQL Server

### 它的作用是什么？

用来存 **结构化数据**，比如：

* 用户信息
* 订单信息
* 商品信息
* 交易记录

### 为什么用 RDS？

因为 AWS 帮你做了很多数据库运维工作，比如：

* 自动备份
* 自动打补丁
* 高可用部署
* 监控
* 故障恢复

### 面试英文说法

> Amazon RDS is a managed relational database service used to store structured data with built-in backup, monitoring, and high availability features.

---

## Amazon S3

S3 = **Simple Storage Service**

中文可以说：

> AWS 的对象存储服务

### 它是干什么的？

主要用来存文件，比如：

* 图片
* 视频
* PDF
* 日志
* 备份文件
* 静态网站资源

### 它的特点

* 容量几乎无限
* 很便宜
* 高耐久
* 扩展性非常强

### 为什么图里 S3 不在 VPC 里面？

因为 **S3 不是部署在你的 VPC 里的资源**。

它是 AWS 提供的托管存储服务。
虽然它不在 VPC 里，但你可以通过：

* IAM 权限
* Bucket Policy
* VPC Endpoint

来安全访问它。

### 面试英文说法

> Amazon S3 is an object storage service used for storing unstructured data such as files, images, logs, and backups with high durability and scalability.

---

# 3. 为什么一个在 VPC 里，一个不在 VPC 里？

这是很多人第一次看图容易懵的点。

## RDS 在 VPC 里

因为它是数据库实例，需要部署在你的网络环境中。

你可以控制：

* 子网
* 安全组
* 是否公网访问
* 路由策略

所以它属于 **VPC 内部资源**。

---

## S3 不在 VPC 里

因为 S3 是 AWS 的区域级托管服务，不是你自己“放进去”的服务器或数据库实例。

也就是说：

* 你不能把一个 S3 bucket 真正部署到你的 VPC 子网里
* 但你可以从 VPC 内部安全访问 S3

---

# 4. CDK 是什么？

CDK = **Cloud Development Kit**

中文可以说：

> 用代码定义云基础设施的工具

传统做法可能是：

* 手动点 AWS Console
* 一个一个创建 VPC、RDS、S3

CDK 的做法是：

* 用代码写出来
* 然后一键部署

比如你写代码定义：

* 一个 VPC
* 一个 RDS
* 一个 S3 Bucket

CDK 就会帮你生成底层 CloudFormation，然后部署到 AWS。

---

## CDK 的好处

### 1）可重复

同样代码可以在不同环境重复部署：

* dev
* test
* prod

### 2）可版本管理

可以放进 Git，团队一起维护。

### 3）可维护

以后架构变更，不用手工点来点去。

### 4）更适合工程化

适合 CI/CD、自动化部署。

### 面试英文说法

> AWS CDK allows us to define cloud infrastructure using code, which improves reusability, maintainability, and deployment automation.

---

# 5. CodeWhisperer 是什么？

CodeWhisperer 是 AWS 的 AI 编码助手。

你可以理解成：

> AWS 版的 AI Copilot

它能帮你：

* 自动补全代码
* 生成 AWS SDK 代码
* 生成 CDK 代码
* 提示安全建议

---

## 它和 CDK 怎么配合？

比如你想创建一个 S3 bucket 或 RDS：

* 你写几行注释
* CodeWhisperer 帮你生成 CDK 代码
* 你再做少量修改
* 最后部署

所以它的作用是：

> 提高开发速度，减少样板代码

### 面试英文说法

> CodeWhisperer is an AI-powered coding assistant that helps generate AWS-related code, including CDK definitions, to improve developer productivity.

# 6. 这张图可能对应什么项目？

这张图本质上是一个很基础的云项目雏形。

---

## 例子 3：数据平台

* 业务配置、小规模事务数据放 **RDS**

# 7. 一个很重要的理解：RDS 和 S3 的区别

| 对比项       | RDS      | S3           |
| --------- | -------- | ------------ |
| 数据类型      | 结构化数据    | 非结构化文件       |
| 典型内容      | 用户、订单、交易 | 图片、视频、日志、备份  |
| 查询方式      | SQL      | 对象路径 / API   |
| 是否在 VPC 内 | 是        | 不是           |
| 适合场景      | 事务型系统    | 文件存储、大规模数据存储 |

# 8. 如果面试官问：为什么要这样设计？

你可以这样回答：

## 中文版

因为不同类型的数据适合不同的存储服务。
结构化、需要事务和查询能力的数据更适合放在 RDS。
而图片、日志、文档这类非结构化数据更适合放在 S3。
同时，把数据库放在 VPC 里可以增强安全性。
使用 CDK 可以让基础设施实现代码化管理，提高部署效率和一致性。

## English version

We separate storage based on data characteristics.
Structured and transactional data is stored in RDS, while unstructured files are stored in S3.
Placing RDS inside a VPC improves security, and using CDK allows us to manage infrastructure as code for better consistency and automation.
