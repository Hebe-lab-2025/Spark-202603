## 这页核心想表达什么

在 AWS CDK 里，**Construct** 就是“用来定义云资源的代码组件”。

你可以把它理解成：

* 你不是手动去 AWS 控制台点点点
* 而是用代码去“描述”要创建什么资源
* 这些描述资源的代码模块，就叫 **Construct**

而 Construct 分成 **L1 / L2 / L3** 三层。

---

# 1. L1 Construct

L1 是最底层。

它几乎就是 **CloudFormation 原始资源的直接映射**。

比如：

* `CfnBucket`
* `CfnVPC`
* `CfnFunction`

这里的 `Cfn` 就是在提示你：
这是直接对应 CloudFormation 的。

### 特点

* 最接近 AWS 原始定义
* 参数很多
* 比较底层
* 灵活，但写起来更麻烦

### 适合什么时候用

* 某个新 AWS 功能，L2/L3 还没封装好
* 你需要非常细粒度控制
* 文档里只有 CloudFormation 属性时

### 例子

```ts
new s3.CfnBucket(this, 'MyBucket', {
  bucketName: 'my-demo-bucket'
});
```

---

# 2. L2 Construct

L2 是最常用的。

它是对 L1 的**面向对象封装**，更好写，也更符合开发者习惯。

比如：

* `s3.Bucket`
* `lambda.Function`
* `dynamodb.Table`

### 特点

* API 更友好
* 自动帮你处理很多默认值
* 代码更简洁
* 更适合日常开发

### 例子

```ts
new s3.Bucket(this, 'MyBucket', {
  versioned: true
});
```

这里你不需要关心太多底层细节，CDK 帮你处理了一部分。

### 你可以这样理解

* **L1 = 原始零件**
* **L2 = 包装好的标准零件**

---

# 3. L3 Construct

L3 是更高层的封装。

它不是只代表一个资源，
而是把**多个资源组合在一起**，变成一个“解决方案组件”。

比如一个常见场景：

* 一个 Lambda
* 一个 API Gateway
* 一些权限配置
* 日志配置

这些如果你每次都自己拼，比较麻烦。
L3 就帮你打包好了。

### 特点

* 更高级抽象
* 封装多个资源
* 更贴近业务场景
* 开发速度更快

### 例子理解

比如你想快速搭一个 “Lambda + API Gateway” 架构：

```ts
new apigateway.LambdaRestApi(this, 'MyApi', {
  handler: myLambda
});
```

这个就是偏 L3 的感觉：
你不是在单独建每一个底层资源，而是在直接建一个完整方案。

### 你可以这样理解

* **L1 = 原材料**
* **L2 = 单个家具**
* **L3 = 一整套装修方案**

---

# 4. 三者最简单对比

| 层级 | 含义                  | 特点          | 适合场景   |
| -- | ------------------- | ----------- | ------ |
| L1 | CloudFormation 原始映射 | 最底层、最灵活、最复杂 | 需要精细控制 |
| L2 | 对单个 AWS 资源的高级封装     | 最常用、最易写     | 日常开发   |
| L3 | 多个资源组合后的高级模式        | 最省事、最贴近业务   | 快速搭方案  |

---

# 5. 面试里怎么说

你可以这样说：

> In AWS CDK, constructs are building blocks for defining cloud infrastructure in code.
> L1 constructs are low-level CloudFormation representations.
> L2 constructs are higher-level abstractions for individual AWS resources.
> L3 constructs are even higher-level patterns that combine multiple resources into reusable solutions.

因为你以后看 CDK 代码时，会经常看到：

* 为什么有些代码一行就能搭完整 API？

答案通常就是：
**它们属于不同层级的 Construct。**

# 这张表怎么理解

## 1. L1 Constructs

### 是什么

L1 是 **Low-level Constructs**，也叫 **CFN Resources**。

意思是：

* 它和 CloudFormation 资源 **1:1 对应**
* 名字一般带 **`Cfn` 前缀**
* 例如：

  * `CfnBucket` 对应 CloudFormation 里的 `AWS::S3::Bucket`

### 特点

* 非常底层
* 参数很多
* 几乎没有智能默认值
* 你要自己配很多细节

### 图里想表达的点

* `CfnBucket`
* `1:1 mapping to CloudFormation Resources`
* VPC 可能要写很多代码，比如图里写的 **282 lines of code for VPC**

### 你可以理解成

**L1 = 原始零件**

你自己一颗螺丝一颗螺丝装。

---

## 2. L2 Constructs

### 是什么

L2 是 **Higher Level Constructs**。

它是在 L1 基础上进一步封装出来的，更符合开发者使用习惯。

比如：

* `s3.Bucket`

### 特点

* 更好用
* 有 **intelligent defaults**（智能默认值）
* 代码更少
* 更接近日常开发

### 图里想表达的点

* `s3.Bucket`
* `Intelligent Defaults`
* 同样是建 VPC，可能只要 **3 lines of code**

### 你可以理解成

**L2 = 半成品组件**

很多底层细节 AWS 已经帮你处理好了，你直接用。

---

## 3. L3 Constructs

### 是什么

L3 是 **Even higher level construct / Pattern**。

也就是更高层的“模式”封装，通常针对一个**具体场景**。

例如图里：

* `aws-apigateway.LambdaRestApi`
* `ecs patterns.ApplicationLoadBalancedFargateService`

### 特点

* 面向具体 use case
* 一次帮你创建多个资源
* 最省代码
* 最适合快速搭架构

### 图里这个例子是什么意思

`aws-apigateway.LambdaRestApi` 不是只建一个 API Gateway。

它背后可能帮你一起处理：

* API Gateway
* Lambda
* REST API
* IAM Role
* 以及其他相关服务

### 你可以理解成

**L3 = 现成套餐**

你不是买零件，也不是买单个组件，而是直接买一整套方案。

---

# 图里最核心的区别

## L1

你要自己处理很多字段，例如：

* BucketName
* Versioning
* 以及更多底层配置

## L2

还是能配这些字段：

* BucketName
* Versioning

但是写法更自然，默认值更智能。

## L3

你甚至不需要先关心底层资源细节，先把**整个业务场景**搭起来。

---

# 最简单类比

## 盖房子类比

* **L1**：水泥、砖头、钢筋，自己一块块搭
* **L2**：门、窗、墙体模块，拿来就能装
* **L3**：整套精装修公寓方案，直接能住

你可以这样讲：

> In AWS CDK, L1 constructs are the lowest-level resources that map directly to CloudFormation.
> L2 constructs are higher-level abstractions with better developer experience and intelligent defaults.
> L3 constructs are patterns built for specific use cases, and they can create multiple related AWS resources together.

---

# 中文面试版简答

> AWS CDK 有三层 construct。
> L1 是最底层，和 CloudFormation 资源一一对应，灵活但代码多。
> L2 是更高一层的封装，有智能默认值，开发中最常用。
> L3 是面向具体场景的高级模式，可以一次创建多个相关资源，开发效率最高。

# 你最该记的结论

## 平时开发怎么选

* **想精细控制底层配置** → 用 **L1**
* **大多数正常开发场景** → 用 **L2**
* **想快速搭一个完整架构模式** → 用 **L3**

## 实际上最常见

日常里通常：

* **L2 最常用**
* **L3 很适合 demo / 快速搭建**
* **L1 只在需要底层细节控制时用**

# 🪜 全流程（超清晰版）

## ✅ Step 1：打开项目

👉 在 VS Code 打开一个新文件夹 + Terminal

---

## ✅ Step 2：创建目录结构

```bash
mkdir infra
mkdir services
cd infra
```

👉 思想：

* `infra` → 基础设施（CDK）
* `services` → 业务代码（Lambda / API）

---

## ✅ Step 3：初始化 CDK 项目

```bash
cdk init app --language typescript
```

👉 会自动生成：

* `lib/` → 写资源
* `bin/` → 入口
* `cdk.json`

---

## ✅ Step 4：引入 AWS 服务模块

```ts
import * as s3 from 'aws-cdk-lib/aws-s3';
```

👉 每个服务一个 module：

* S3 → aws-s3
* Lambda → aws-lambda
* DynamoDB → aws-dynamodb

---

## ✅ Step 5：定义资源（核心🔥）

```ts
new s3.Bucket(this, 'MyBucket', {
  versioned: true
});
```

👉 三个关键点：

* `this` → scope（在哪个 stack）
* `'MyBucket'` → logical ID
* `{}` → 配置参数

---

## ✅ Step 6：构建（可选）

```bash
npm run build
```

👉 TypeScript 编译 → JS

---

## ✅ Step 7：Bootstrap（只做一次）

```bash
cdk bootstrap
```

👉 本质：

* 给 AWS 账号准备 CDK 环境（S3 bucket / IAM role）

---

## ✅ Step 8：生成 CloudFormation

```bash
cdk synth
```

👉 输出：
👉 CloudFormation JSON 模板

---

## ✅ Step 9：部署 🚀

```bash
cdk deploy
```

👉 真正创建 AWS 资源！

* 创建 CDK 运行需要的 S3 / IAM 环境
