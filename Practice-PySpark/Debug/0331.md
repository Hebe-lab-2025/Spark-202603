## 4. Java Lambda 代码

### `Handler.java`

```java
package com.example;

import com.amazonaws.services.lambda.runtime.Context;
import com.amazonaws.services.lambda.runtime.RequestHandler;
import software.amazon.awssdk.services.dynamodb.DynamoDbClient;
import software.amazon.awssdk.services.dynamodb.model.AttributeValue;
import software.amazon.awssdk.services.dynamodb.model.PutItemRequest;

import java.util.HashMap;
import java.util.List;
import java.util.Map;

public class Handler implements RequestHandler<Map<String, Object>, Response> {

    private final DynamoDbClient dynamoDbClient = DynamoDbClient.create();
    private final String tableName = System.getenv("TABLE_NAME");

    @Override
    public Response handleRequest(Map<String, Object> event, Context context) {
        context.getLogger().log("Received event: " + event);

        try {
            Object recordsObj = event.get("Records");
            if (!(recordsObj instanceof List)) {
                return new Response(400, "No Records found in event");
            }

            List<?> records = (List<?>) recordsObj;

            for (Object recordObj : records) {
                if (!(recordObj instanceof Map)) {
                    continue;
                }

                Map<?, ?> record = (Map<?, ?>) recordObj;
                Map<?, ?> s3 = (Map<?, ?>) record.get("s3");
                Map<?, ?> bucket = (Map<?, ?>) s3.get("bucket");
                Map<?, ?> object = (Map<?, ?>) s3.get("object");

                String bucketName = String.valueOf(bucket.get("name"));
                String fileName = String.valueOf(object.get("key"));
                String fileSize = String.valueOf(object.getOrDefault("size", "0"));

                Map<String, AttributeValue> item = new HashMap<>();
                item.put("fileName", AttributeValue.builder().s(fileName).build());
                item.put("bucketName", AttributeValue.builder().s(bucketName).build());
                item.put("fileSize", AttributeValue.builder().n(fileSize).build());

                PutItemRequest putItemRequest = PutItemRequest.builder()
                        .tableName(tableName)
                        .item(item)
                        .build();

                dynamoDbClient.putItem(putItemRequest);
            }

            return new Response(200, "File metadata saved successfully");
        } catch (Exception e) {
            context.getLogger().log("Error: " + e.getMessage());
            return new Response(500, "Internal Server Error: " + e.getMessage());
        }
    }
}
```

## 7. CDK Stack

### `infra/lib/infra-stack.ts`

```ts
import * as cdk from 'aws-cdk-lib';
import { Construct } from 'constructs';
import * as s3 from 'aws-cdk-lib/aws-s3';
import * as lambda from 'aws-cdk-lib/aws-lambda';
import * as dynamodb from 'aws-cdk-lib/aws-dynamodb';
import * as s3n from 'aws-cdk-lib/aws-s3-notifications';

export class InfraStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    const table = new dynamodb.Table(this, 'FileMetadataTable', {
      partitionKey: {
        name: 'fileName',
        type: dynamodb.AttributeType.STRING,
      },
      billingMode: dynamodb.BillingMode.PAY_PER_REQUEST,
      removalPolicy: cdk.RemovalPolicy.DESTROY,
    });

    const bucket = new s3.Bucket(this, 'UploadBucket', {
      removalPolicy: cdk.RemovalPolicy.DESTROY,
      autoDeleteObjects: true,
    });

    const fileProcessorFunction = new lambda.Function(this, 'JavaFileProcessorFunction', {
      runtime: lambda.Runtime.JAVA_17,
      handler: 'com.example.Handler::handleRequest',
      code: lambda.Code.fromAsset('../services/lambda-java/target'),
      memorySize: 1024,
      timeout: cdk.Duration.seconds(30),
      environment: {
        TABLE_NAME: table.tableName,
      },
    });

    table.grantWriteData(fileProcessorFunction);

    bucket.addEventNotification(
      s3.EventType.OBJECT_CREATED,
      new s3n.LambdaDestination(fileProcessorFunction)
    );

    new cdk.CfnOutput(this, 'BucketName', {
      value: bucket.bucketName,
    });

    new cdk.CfnOutput(this, 'TableName', {
      value: table.tableName,
    });

    new cdk.CfnOutput(this, 'LambdaName', {
      value: fileProcessorFunction.functionName,
    });
  }
}
```

# B. 面试问答 + 10 个 follow-up

---

## 1. 1-minute answer

> I built a small event-driven AWS project using CDK.
> I provisioned an S3 bucket, a Java Lambda function, and a DynamoDB table.
> When a user uploaded a file to S3, it automatically triggered the Lambda function.
> The Lambda extracted file metadata such as the bucket name, file key, and file size, and stored that data in DynamoDB.
> This project helped me better understand infrastructure as code, AWS service integration, and event-driven backend design.

---

## 2. Why did you build this project?

> I wanted a hands-on way to learn AWS event-driven architecture instead of only reading documentation.
> This project let me practice core services like S3, Lambda, DynamoDB, and CDK together in one end-to-end workflow.

---

## 3. Why did you use CDK instead of creating resources manually?

> I used CDK because it makes infrastructure reproducible and easier to maintain.
> Instead of clicking in the console, I could define the whole environment in code, version it in Git, and redeploy it consistently.

---

## 4. Why did you choose S3 + Lambda?

> S3 and Lambda are a natural fit for event-driven processing.
> Once a file is uploaded, S3 can automatically trigger Lambda, which removes the need for polling and keeps the design simple.

---

## 5. Why did you use DynamoDB?

> I only needed to store simple metadata, so DynamoDB was a good fit because it is serverless, scalable, and easy to integrate with Lambda.

---

## 6. What was the most important technical part?

> The most important part was wiring the services together correctly.
> I had to define the infrastructure in CDK, make sure S3 notifications triggered Lambda, and make sure Lambda had permission to write to DynamoDB.

---

## 7. What challenges did you face?

> The main challenges were packaging the Java Lambda correctly, setting up permissions, and verifying whether the trigger chain actually worked end to end.

---

## 8. How did you debug it?

> I used a step-by-step approach.
> First I confirmed the stack deployed successfully.
> Then I uploaded a test file to S3.
> After that I checked CloudWatch logs to verify Lambda execution.
> Finally, I checked DynamoDB to confirm the record was written.

---

## 9. What did you learn?

> I learned how infrastructure as code works in practice, how AWS event-driven systems are connected, and how to debug service integrations rather than just application code.

---

## 10. If you improved it, what would you add?

> I would add file type validation, better error handling, an SNS notification layer, and maybe an API Gateway endpoint to query processed file records.

# 10 follow-up questions + sample answers

---

## Follow-up 1: Why Java Lambda instead of Python?

> I chose Java because it aligns better with my backend interview focus.
> It also helped me practice AWS integration using a language I use more often in backend development.

---

## Follow-up 2: What is the role of the Lambda handler?

> The handler is the entry point of the Lambda function.
> It receives the S3 event payload, extracts the file information, and then writes that information into DynamoDB.

---

## Follow-up 3: What permissions were needed?

> Lambda needed permission to write to DynamoDB, and S3 needed permission to invoke Lambda through the event notification setup.

---

## Follow-up 4: What does `cdk synth` do?

> `cdk synth` converts CDK code into a CloudFormation template.
> It is useful for checking what will be deployed before actually creating resources.

---

## Follow-up 5: What does `cdk bootstrap` do?

> `cdk bootstrap` prepares the AWS environment for CDK deployment, such as creating the resources CDK needs behind the scenes in that account and region.

---

## Follow-up 6: How would you handle duplicate file names?

> I would redesign the DynamoDB key structure, for example by adding upload timestamp or object version as a sort key, so multiple files with the same name can be stored safely.

---

## Follow-up 7: How would you make the system more production ready?

> I would add input validation, structured logging, DLQ or retry handling, monitoring alarms, least-privilege IAM policies, and safer removal policies instead of destroy.

---

## Follow-up 8: What happens if Lambda fails?

> The file would still exist in S3, but the metadata might not be written to DynamoDB.
> To improve reliability, I would add better retry handling and maybe route failures to a dead-letter destination.

---

## Follow-up 9: Why is this event-driven?

> It is event-driven because the system reacts automatically to an S3 upload event instead of waiting for a scheduled job or a manual trigger.

---

## Follow-up 10: What is the biggest value of this project for interviews?

> The biggest value is that it shows both backend coding and cloud architecture thinking.
> I can talk about implementation, infrastructure as code, AWS integration, and debugging in one project.
