# ⚡ 二、两种类型（面试必考🔥）

## 1️⃣ Batch Ingestion（批处理）

👉 一次性导入

**例子：**

* 每天导入日志
* 数据库定时 dump

**AWS 服务：**

* AWS Glue
* AWS Data Pipeline

👉 面试一句话：

> Batch = high throughput, high latency

## 2️⃣ Streaming Ingestion（实时🔥）

👉 数据实时进来

**例子：**

* 用户点击流
* IoT 数据
* 订单流

**AWS 服务：**

* Amazon Kinesis
* Amazon Kinesis Data Firehose

👉 面试一句话：

> Streaming = low latency, real-time processing

# 🔥 三、核心 AWS 服务（必须会区分）

## 🟢 Kinesis vs Firehose（高频🔥）

### Amazon Kinesis

* 自己写 consumer（复杂）
* 实时处理（毫秒级）
* 灵活

👉 用于：

* 实时 analytics
* fraud detection

### Amazon Kinesis Data Firehose

* 全自动（不用写代码）
* 自动写入 S3 / Redshift
* 延迟稍高

👉 用于：

* log ingestion（最常见🔥）


👉 面试标准答案：

> Kinesis = control
> Firehose = simplicity


## 🟡 SQS / SNS（解耦🔥）

* Amazon SQS → queue（缓冲）
* Amazon SNS → pub/sub

👉 作用：

> 防止系统被打爆（decouple）

# 🧪 四、典型架构（面试直接说🔥）

## 🔥 实时 ingestion pipeline

```
Producer → Kinesis → Lambda → S3
```

👉 升级版：

```
Kinesis → Firehose → S3 → Glue → Athena
```


# 🎯 五、面试高频问题（Amazon风格）

## ❓ Q1：Design a data ingestion system

👉 标准回答结构：

**1️⃣ Source**

* mobile / logs / IoT

**2️⃣ Ingestion**

* Kinesis / Firehose

**3️⃣ Buffer**

* SQS（optional）

**4️⃣ Storage**

* S3（data lake）


👉 一句话总结：
> I would use Kinesis for streaming ingestion and store data in S3 for scalability.


## ❓ Q2：How do you handle spikes?

👉 关键词：

* auto scaling（Kinesis shard）
* buffer（SQS）
* retry


## ❓ Q3：How to guarantee reliability?

👉 答：

* retry
* DLQ（SQS）
* checkpoint


## ❓ Q4：Exactly once vs at least once

👉 标准：

* Kinesis → at least once
* 用 idempotent 解决重复


# 🧠 六、考试重点（直接考🔥）

👉 你要背：

* Firehose 自动 delivery 到 S3
* Kinesis shard = throughput
* SQS = decouple
* Glue crawler = schema

## ✅ 立刻实战（强烈推荐）

做这个项目：

👉 ingestion 核心三句话：

# 🧠 一、S3 是什么？

👉 本质：

* 存文件（不是数据库）
* 无限扩展
* 高可用（11个9 durability）

📌 举例：

* 图片
* 日志
* CSV / Parquet 数据
* Data Lake 数据

---

# 🧠 二、S3 的结构（超重要🔥）

👉 你必须会这个：

```
Bucket（桶） → 类似文件夹根目录
  ├── object（对象）→ 文件
  ├── key → 文件路径
```

📌 举例：

```
s3://my-bucket/data/2025/03/file.csv
```

👉

* bucket = my-bucket
* key = data/2025/03/file.csv

---

# 🧠 三、为什么 S3 在数据工程里这么重要？

👉 因为：

🔥 **所有数据服务都围绕 S3**

| 服务       | 怎么用 S3   |
| -------- | -------- |
| Glue     | 从 S3 读数据 |
| Athena   | 查 S3     |
| Redshift | COPY S3  |
| EMR      | 处理 S3    |
| Kinesis  | 写入 S3    |

👉 👉 S3 = 数据中心（Data Lake）

---

| Standard              | 常用数据 |
| IA（Infrequent Access） | 不常访问 |
| Glacier               | 归档   |
| Glacier Deep Archive  | 超便宜  |

👉 面试一句话：

> 访问少 → 用 Glacier 省钱

```
30天 → IA
```

👉 用来：

* 降成本 💰

👉 防止误删

| ----------- | ------- |
| SSE-S3      | AWS管理   |
| SSE-KMS     | 更安全（常考） |
| Client-side | 自己加密    |

👉 考试默认选：
**SSE-KMS**

## 5️⃣ 权限控制

👉 两种：


👉 面试：

# 🧠 一、基础概念（必问）

---

## ❓1. What is Amazon S3?

👉 答：

> S3 is a highly scalable and durable object storage service used as a data lake in data engineering.

---

## ❓2. What are bucket and object?

👉 答：

> A bucket is a container, and objects are files stored inside it.

---

## ❓3. Why is S3 important for data engineering?

👉 答：

> It provides cheap, scalable storage for large-scale analytics workloads.

---

## ❓4. S3 vs EBS vs EFS?

👉 答：

| 服务  | 用途                        |
| --- | ------------------------- |
| S3  | object storage（data lake） |
| EBS | block storage（EC2）        |
| EFS | shared file system        |

👉 一句话：

> S3 is for big data, not for low-latency disk access.

---

# ⚡ 二、性能 & 优化（高频🔥）

---

## ❓5. How do you optimize query performance on S3?

👉 答：

> Use partitioning, columnar formats like Parquet, and compression.

---

## ❓6. What is partitioning in S3?

👉 答：

> Organizing data by keys like date to reduce scan size.

---

## ❓7. What is the small file problem?

👉 答：

> Too many small files increase overhead and slow down queries.

👉 解决：

> Merge files using Spark.

---

## ❓8. Why use Parquet instead of JSON?

👉 答：

> Parquet is columnar, so it reduces scan cost and improves performance.

# AWS S3 - Storage

这页在讲 **AWS 的 S3 存储服务**，它属于 **Data Management（数据管理）** 里面很核心的一部分。

## 1. S3 是什么？

**S3 = Simple Storage Service**

它是 AWS 提供的 **对象存储（object storage）** 服务。

你可以把它理解成：

* **bucket** = 一个大文件夹 / 容器
* **object** = 存在 bucket 里的文件

所以这页写的：

> Buckets (containers for storage) and objects (files)

意思就是：

* **Bucket**：用来装数据的“桶”
* **Object**：真正存进去的文件，比如：

  * 图片
  * PDF
  * 日志文件
  * CSV
  * Parquet
  * 视频
  * 备份文件

---

## 2. bucket 和 object 的关系

可以这样理解：

* 你先创建一个 **bucket**
* 再把文件上传进去
* 上传进去的每个文件，就是一个 **object**

比如：

你建了一个 bucket：

```text
my-data-bucket
```

里面可以放：

```text
sales.csv
logs/2026/03/app.log
images/photo1.png
backup/db.dump
```

这里：

* `my-data-bucket` 是 bucket
* `sales.csv`、`app.log`、`photo1.png` 都是 object

---

## 3. 为什么叫 object，不直接叫 file？

因为在 S3 里，AWS 不只是把它当普通文件。

每个 object 一般包含：

* **数据本身**
* **key（路径/名字）**
* **metadata（元数据）**

比如一个 object 可能有：

* 文件名
* 大小
* 上传时间
* 存储类型
* 加密信息

所以 AWS 用 **object** 这个词，而不是只说 file。

---

## 4. “Each bucket is created in a specific region” 是什么意思？

这句话意思是：

**每个 bucket 创建时，都要指定一个 AWS 区域（region）**

例如：

* us-east-1
* us-west-2
* ap-northeast-1

也就是说：

你的 bucket 不是“全世界随便放”的，
而是会放在你创建时选定的那个区域里。

### 为什么这点重要？

因为它会影响：

* **延迟（latency）**
* **合规（compliance）**
* **成本（cost）**
* **数据驻留（data residency）**

比如：

* 你的用户主要在美国西部，可能更适合 `us-west-2`
* 如果公司要求数据存在加拿大，就可能选加拿大 region
* 不同 region 的价格也可能不一样

---

## 5. 这页两张截图在表达什么？

### 第一张图

展示的是 **某一个 bucket 里面有哪些 objects**

你可以看到 bucket 里会有：

* 文件夹样式的路径
* 不同文件
* 最后修改时间
* 文件大小
* 存储类型

意思就是：
**S3 可以像文件管理器一样去看 bucket 里的内容。**

### 第二张图

展示的是 **账户下有哪些 buckets**

并且每个 bucket 会显示：

* bucket 名字
* AWS region
* 访问设置
* 创建时间

意思就是：
**你可以创建多个 bucket，而且每个 bucket 属于某个特定 region。**

因为 key 不一样。

* 日志存储
* 灾难恢复

> S3 is commonly used for backup due to its high durability.

👉 S3 可以直接托管：

* HTML
* CSS
* JS

👉 架构：

```text
S3 + CloudFront（CDN）
```

👉 面试答：

> S3 can host static websites with high availability.

---

👉 用：

* Glacier
* lifecycle policy

👉 面试答：

> Move cold data to Glacier to reduce cost.

* 支持恢复

* Standard → IA → Glacier

👉 面试答：

> Lifecycle policies optimize cost over time.

👉 类型：

* CRR（跨 region）
* SRR（同 region）

👉 面试答：

> Replication improves disaster recovery.

---

## ✅ 4️⃣ Encryption（安全）

👉 类型：

* SSE-S3
* SSE-KMS

# 🧠 二、6种存储类型（考试重点）

---

## 1️⃣ S3 Standard（默认）

👉 用途：

* 经常访问的数据

👉 特点：

* 高可用（99.99%）
* 低延迟

✅ 场景：

* 网站图片
* 热数据

---

## 2️⃣ S3 Intelligent-Tiering（很爱考🔥）

👉 用途：

* **访问频率不确定**

👉 特点：

* 自动在不同层之间切换

✅ 场景：

* 不知道用户会不会访问的数据

👉 面试一句话：

> 自动优化成本

---

## 3️⃣ S3 Standard-IA（Infrequent Access）

👉 用途：

* 偶尔访问，但要**快速读取**

👉 特点：

* 比 Standard 便宜
* 但读取要收费

✅ 场景：

* 备份数据

---

## 4️⃣ S3 One Zone-IA

👉 用途：

* 不重要数据

👉 特点：

* 只在一个 AZ（更便宜）
* 风险更高

⚠️ 场景：

* 可重建数据

---

## 5️⃣ S3 Glacier

👉 用途：

* 长期归档

👉 特点：

* 读取需要几分钟到几小时

✅ 场景：

* 合规数据
* 日志归档

---

## 6️⃣ S3 Glacier Deep Archive（最便宜🔥）

👉 用途：

* 超长期存储

👉 特点：

* 读取需要12小时

✅ 场景：

* 7年归档

---

# 🧠 三、Durability vs Availability（必考区别🔥）

---

## 🔥 Durability（耐久性）

👉
**数据丢失概率**

* S3 = 99.999999999%（11个9）
  👉 基本不会丢

---

## 🔥 Availability（可用性）

👉
**能不能访问到数据**

* Standard：99.99%
* IA：99.9%

👉 👉 可能“暂时访问不到”，但不会丢

---


👉 面试关键词：

> cost optimization

# 🧠 五、Versioning（必考）

👉 功能：

# 🧠 六、考试秒选技巧（最重要🔥）

| --------- | ------------------- |
| 经常访问      | Standard            |

| 超长期归档     | Deep Archive        |

# 🧠 二、10道综合场景题（高频🔥）

---

## 1️⃣ 查询 S3 数据（不想建仓库）

👉 场景：
数据在 S3，想直接用 SQL 查询

✅ 答案：**Athena**

👉 理由：
Serverless + 不需要加载数据

---

## 2️⃣ S3 → Redshift 自动ETL

👉 场景：
每天把 S3 数据加载到 Redshift

✅ 答案：**Glue**

👉 理由：
ETL 自动化

---

## 3️⃣ 查询慢（Athena 很慢）

👉 场景：
Athena 查询 S3 很慢

✅ 答案：
👉 **Partition + Parquet**

👉 理由：
减少扫描数据量

---

## 4️⃣ 实时数据分析

👉 场景：
实时日志处理

✅ 答案：**Kinesis**

👉 理由：
流式数据

---

## 5️⃣ 高性能 BI 报表

👉 场景：
dashboard + 高并发查询

✅ 答案：**Redshift**

👉 理由：
OLAP优化

---

## 6️⃣ 只偶尔查询历史数据

👉 场景：
偶尔查历史数据

✅ 答案：**Athena**

👉 理由：
便宜 + 按查询付费

---

## 7️⃣ 数据 schema 管理

👉 场景：
S3数据需要统一 schema

✅ 答案：**Glue Data Catalog**

---

## 8️⃣ 数据 pipeline 编排

👉 场景：
多个步骤流程控制

✅ 答案：**Step Functions**

👉 理由：
只负责 orchestration

---

## 9️⃣ ETL + 不想管理服务器

👉 场景：
自动数据处理

✅ 答案：**Glue**

---

## 🔟 大规模复杂计算（PB级）

👉 场景：
复杂 Spark 任务

✅ 答案：**EMR**

👉 理由：
更灵活

---

👉

---

👉 分析：
