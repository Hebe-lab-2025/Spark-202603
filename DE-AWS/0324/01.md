# 🧾 PySpark 高频面试坑（Top 10）

## 1️⃣ 忘记调用 action（最常见）

```python
df.show      # ❌
df.show()    # ✅
```

👉 不调用 action → 不会执行（lazy execution）

---

## 2️⃣ 误以为 transformation 会执行

```python
df.filter(df.age > 18)
```

👉 ❌ 不会执行
👉 ✅ 必须配合 action：

```python
df.filter(df.age > 18).show()
```

---

## 3️⃣ 多次 action 导致重复计算

```python
df.count()
df.show()
```

👉 ❌ 会执行两次 job（性能差）

✅ 优化：

```python
df.cache()
df.count()
df.show()
```

---

## 4️⃣ collect() 导致 OOM（高频面试问）

```python
df.collect()
```

👉 ❌ 把所有数据拉到 driver（容易炸内存）

✅ 替代：

```python
df.show()
df.take(10)
```

---

## 5️⃣ groupByKey vs reduceByKey（必考）

```python
rdd.groupByKey()   # ❌ 慢，占内存
rdd.reduceByKey()  # ✅ 推荐
```

👉 groupByKey 会 shuffle 所有数据
👉 reduceByKey 会提前聚合（map-side combine）

---

## 6️⃣ repartition vs coalesce（经典）

```python
df.repartition(100)  # shuffle
df.coalesce(1)       # 不shuffle（减少分区）
```

👉 面试回答：

* repartition → **增加/减少分区 + shuffle**
* coalesce → **只减少分区 + 无shuffle（更快）**

---

## 7️⃣ join 导致数据倾斜（data skew）

👉 某个 key 特别大 → executor 挂掉

✅ 解决：

* salting（加随机数）
* broadcast join
* skew join optimization

---

## 8️⃣ broadcast join 忘记用

```python
df1.join(df2, "id")
```

👉 小表 join 大表时：

```python
from pyspark.sql.functions import broadcast
df1.join(broadcast(df2), "id")
```

👉 🚀 避免 shuffle（高频优化点）

---

## 9️⃣ UDF 性能差（面试必问）

```python
from pyspark.sql.functions import udf
```

👉 ❌ Python UDF 慢（跨 JVM）

✅ 优先：

* built-in functions（最快）
* pandas UDF（vectorized）

---

## 🔟 cache 不释放（隐藏坑）

```python
df.cache()
```

👉 ❌ 占内存不释放

✅ 正确：

```python
df.unpersist()
```

# 🧠 一页速记表（直接背）

## 🔥 核心模型

👉 **Lazy Evaluation**

```
Transformation → 不执行
Action → 触发执行（job）
```

---

## 🔥 Execution Flow（面试必说）

```
Driver → DAG → Stage → Task → Executor
```

---

## 🔥 高频对比

| 场景   | 错误            | 正确                    |
| ---- | ------------- | --------------------- |
| 执行   | `df.show`     | `df.show()`           |
| 大数据  | `collect()`   | `take()`              |
| 聚合   | `groupByKey`  | `reduceByKey`         |
| 分区   | `repartition` | `coalesce`            |
| join | 普通 join       | broadcast join        |
| UDF  | Python UDF    | built-in / pandas UDF |

---

## 🔥 性能优化口诀（直接背）

👉 **少 shuffle + 少 collect + 多 cache + 用 broadcast**

---

## 🔥 一句话面试总结（万能句）

👉
**Spark performance issues usually come from shuffle, data skew, and unnecessary actions. Optimization focuses on reducing data movement and leveraging in-memory computation.**

---

## 🎯 加分回答（面试官会很喜欢）

👉

* Use **broadcast join** to avoid shuffle
* Use **reduceByKey instead of groupByKey**
* Avoid **collect() on large datasets**
* Cache reused DataFrames
* Handle **data skew with salting**

# 2. Local 模式

## 原理

Local 模式本质上是 **一个进程 + 内部线程** 来模拟 Spark 运行环境。适合学习和本地调试。

## 交互式环境

### `bin/pyspark`

作用：启动 Python 交互式 Spark 环境。

文中示例：

```python
sc.parallelize([1,2,3,4,5]).map(lambda x: x + 1).collect()
```

这段代码做了三件事：

1. `parallelize([1,2,3,4,5])`：把本地集合变成 RDD
2. `map(lambda x: x + 1)`：对每个元素加 1
3. `collect()`：把结果拉回 Driver。

### `bin/spark-shell`

作用：启动 Scala 交互式环境。

文中示例：

```scala
sc.parallelize(Array(1,2,3,4,5)).map(x=> x + 1).collect()
```

### `bin/spark-submit`

作用：正式提交 Spark 程序。文中示例：

```bash
bin/spark-submit /export/server/spark/examples/src/main/python/pi.py 10
```

其中 `10` 是程序参数，数字越大，PI 估算越精确。

## Web UI

文中说明 Spark 任务运行时会绑定到 `4040` 端口；如果被占用会顺延到 `4041`、`4042`。这个端口是任务监控页面。

# 9. RDD 核心

## RDD 是什么

文中总结的关键词是：

* 不可变
* 可分区
* 并行计算
* 弹性分布式数据集。

## RDD 五大特性

文中给出的理解：

* RDD 有分区
* 方法作用在所有分区上
* RDD 之间有依赖关系
* KV 类型的 RDD 有分区器
* 分区规划会尽量靠近数据所在服务器。

## RDD 创建

### 方式一：并行化集合

文中示例思想：

```python
sc.parallelize([1,2,3])
```

补充细节：

* 把本地 Python 集合转成分布式 RDD
* 在 `local[*]` 模式下，如果不指定分区数，默认通常按 CPU 核数决定。文中有明确说明。

### 方式二：读取外部数据

文中示例思想：

```python
sc.textFile("path")
```

作用：

* 从文件系统读取文本
* 每一行变成一个元素。

# 10. RDD Transformation 详细整理

## `map`

对每个元素做一对一转换。博客在 Local 章节和算子章节都体现了这个思路。

典型形式：

```python
rdd.map(lambda x: x + 1)
```

## `flatMap`

先映射再打平，常用于分词。

典型思路：

```python
rdd.flatMap(lambda x: x.split(" "))
```

## `reduceByKey`

按 key 聚合，带预聚合，性能通常优于 `groupByKey`。文中多次强调。

典型思路：

```python
rdd.reduceByKey(lambda a, b: a + b)
```

## `mapValues`

只改 value，不改 key。

## `groupBy`

按规则分组。文中列为高频转换算子。

## `filter`

按条件过滤。

## `distinct`

去重。

## `union`

合并两个 RDD。

## `join`

按 key 连接两个 KV RDD。

## `intersection`

取交集。

## `glom`

把每个分区的数据收成一个列表，方便查看分区内容。

## `groupByKey`

按 key 分组，但不做预聚合。文中提醒：

* `groupByKey` 只保留同组的值
* `groupBy` 还保留 key 规则语义
* 聚合场景一般不如 `reduceByKey`。

## `sortBy`

通用排序。文中特别提醒：

* 多分区排序只能保证局部有序
* 如果要全局有序，分区数最好设为 1。

## `sortByKey`

按 key 排序。

# 12. 分区相关算子

## `mapPartitions`

文中重点：

* 不是节省 CPU
* 主要是减少函数调用和网络管道 IO 开销
* 所以很多场景比 `map` 更快。

## `foreachPartition`

和 `mapPartitions` 区别：

* `mapPartitions` 有返回值，属于 Transformation
* `foreachPartition` 没返回值，属于 Action。

## `partitionBy`

给 KV RDD 指定分区器。

## `repartition`

本质上是重新分区，底层会走 shuffle。

## `coalesce`

文中提到：

* 有安全机制
* 如果增加分区但 `shuffle=False`，分区不会真的增加。

## `repartition` 和 `coalesce` 的关系

文中明确写到：

* `repartition` 底层调用的是 `coalesce`
* 只是默认 `shuffle=True`。

# 13. 高频面试点：`groupByKey` vs `reduceByKey`

文中结论很明确：

* `reduceByKey` 自带聚合逻辑
* `groupByKey` 不带聚合逻辑
* 如果是聚合场景，`reduceByKey` 更好
* 因为它可以先局部聚合，再 shuffle，网络 IO 更小。

# 14. 持久化：Cache vs Checkpoint

## Cache

文中总结：

* 轻量级
* 可存内存和磁盘
* 分散存储
* 保留血缘
* 性能更好。

文中总结：

* 重量级
* 集中存储
* 通常落到 HDFS
* 不保留血缘
* 更安全，但慢。
